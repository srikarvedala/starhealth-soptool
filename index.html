Processing Steps
Follow these steps in order for every CYBLE leaked credentials alert.
01
Open CYBLE alert in Cortex Console
Check source, title, and domain of the alert
02
Verify exposed data ownership
Determine if data belongs to employee / agent / hospital / customer
03
Check for sunsetted domains
If alert source is agent.starhealth.in, spp.starhealth.in, or shpv.starhealth.in → mark as False Positive and stop
04
Generate ticket on Nexus Platform
Refer section 5.0 for ticket creation procedure
05
Segregate alerts by user classification
Use classification rules to categorise each alert
06
Share analysis with On-site SOC team
For password policy validation
07
Mark alert as Under Review
Update status in Cortex Console
08
Resolve alert after SOC confirmation
Do not resolve before receiving confirmation
09
Ensure credential refresh / password reset
For the concerned individual
10
Document proof and close the ticket
Attach all evidence before closing
Classification Rules
Pattern reference for categorising CYBLE alert users.
E
Employee
› Email domain is @starhealth.in
› Email domain is @starinsurance.in
› Username contains SH code e.g. SH12345
A
Agent
› Username contains BA code e.g. BA0000
› Username in SH:BA format e.g. SH:BA12345
H
Hospital
› Email where username part contains "hospital" e.g. abchospital@gmail.com
› Username starts with Hos
› Username starts with STH
C
Customer
› Usernames containing numbers (not SH/BA)
› Generic names or usernames with spaces
› Usernames with special characters
› Any @gmail.com or non-employee email
› Anything else not covered above
FP
False Positive — Sunsetted Domains
agent.starhealth.inspp.starhealth.inshpv.starhealth.in
› If any CYBLE alert is triggered from the above domains, mark immediately as False Positive. No further action required.
